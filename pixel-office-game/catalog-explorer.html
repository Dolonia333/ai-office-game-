🔴 Root Causes & Fixes

Four fundamental disconnects in the current system.

1. No layer / depth rules
Depth is a hand-coded number per object (1.2, 1.5, 1.6, 2…). There's no rule like "decor always above furniture" or "floor always at 0". When you add a new object you have to guess the right depth, and things draw in the wrong order constantly.

The tile system uses a completely different depth convention — corridor floor is 0.1, clinic floor is 0.5, walls are 5. Furniture is 1.5. Characters are somewhere around 2–3. There's no single source of truth.
✅ Fix: define a layer enum (floor → 0, rug → 0.5, furniture → 1.5, decor → 2, character → 3, wall → 5, ui → 100) and derive depth from layer name.
2. Tiles and furniture are completely disconnected
tiles-catalog.json handles floors and walls. furniture_catalog*.json handles props. They use different ID schemes, different coordinate systems, and there's no way to ask "what tile should go under this desk?" or "can this furniture be placed on top of this floor type?"

The tilesheets-config only defines index ranges by category — it has no semantic tag like "walkable", "solid", "water-only", or "connects_to: wall".
✅ Fix: add layer, tile_repeat, connects_to fields to both systems. A desk could say allowed_floors: ["floor_office_wood"].
3. 49,168 singles with no placement rules
The master catalog has entries for all your LimeZu singles, but zero of them have placement info. You can't use them in any layout without manually writing a furniture_catalog entry for each one.

The semantics guessing (extract_semantics()) is also unreliable — it only detects a handful of keyword categories and defaults everything else to "type": "furniture". Most entries have wrong or missing types.
✅ Fix: add a verified: false field and the Sprite Tagger tool (Tab 4) to confirm/override auto-guessed fields. Then generate placements from verified singles.
4. No snap-to-grid or collision metadata
Every object's collision is hard-coded in office-scene.js as a 16px-tall invisible rectangle at the sprite's feet. There's no way for the catalog to say "this desk is 2 tiles wide and 1 deep" or "this wall blocks all movement". Furniture size is not connected to the tile grid.

Decor (monitors, lamps) correctly has no physics body — but this is determined by checking def.type !== 'decor', which breaks if type is wrong.
✅ Fix: add snap_to_grid (tile dimensions), collision (none/feet/full), and footprint (w×h in tiles) to each catalog entry.

📐 Unified Sprite Entry Schema

A single JSON format that could replace all 6 catalogs. Existing fields are preserved; purple fields are new.

id
string — required
Unique stable identifier. Never changes once assigned.
"desk_pod_black_32"
source_type
"sheet_slice" | "single_file"
How the sprite pixels are retrieved at runtime.
"sheet_slice"
sheet
string — for sheet_slice
Phaser texture key that must be preloaded.
"mo_black_shadow_32"
url_path
string — for single_file
Relative path from pixel game stuff root.
"assets/Modern_Office.../Singles_1.png"
rect
{ x, y, w, h } — pixels
Exact crop rectangle on the sheet.
{ "x": 0, "y": 0, "w": 64, "h": 64 }
origin
"bottom" | "center"
Anchor point for placement. "bottom" = setOrigin(0.5, 1).
"bottom"
layer
enum — NEW ✦
Controls draw order. One of: floor, rug, furniture, decor, character, wall, ui.
"furniture"
collision
"none" | "feet" | "full" — NEW ✦
"feet" = 16px invisible box at bottom (current default). "full" = entire bounding box. "none" = no physics.
"feet"
snap_to_grid
boolean — NEW ✦
If true, placement x/y should snap to 32px tile grid.
true
footprint
{ w, h } tiles — NEW ✦
How many tiles this sprite occupies. Used for grid placement and collision sizing.
{ "w": 2, "h": 2 }
tile_repeat
boolean — NEW ✦
If true, this sprite is a tile that should be stamped in a repeating tileSprite (floors, walls, rugs).
false
connects_to
string[] — NEW ✦
IDs or categories this can connect to. Walls say "wall". Floors say "floor_office_wood".
["wall_interior"]
allowed_floors
string[] — NEW ✦
Floor categories this object can sit on. Empty = any floor.
["floor_office_wood"]
variants
string[] — NEW ✦
Color/material variants that exist. Used for theming rooms.
["black", "white", "wood"]
type
string — existing, clarified
Semantic type. Use: surface, seat, decor, furniture, floor, wall, character, terrain, water, storage.
"surface"
action
string | null — NEW ✦
What happens when a character interacts: seat, use_computer, sleep, use_it, drink, print.
"seat"
animation
object | null — NEW ✦
For animated singles: { frameW, frameH, frameCount, fps, loop }.
{ "frameW": 16, "fps": 8, "loop": true }
verified
boolean — NEW ✦
Whether a human has confirmed this entry is correct. Auto-generated entries start as false.
false
label
string — NEW ✦
Human-readable display name for the asset browser and tagger.
"Black shadow office desk (32px)"
tags
string[] — NEW ✦
Free-form searchable tags. Used by asset browser and proc-gen to find sprites.
["office", "desk", "work"]
